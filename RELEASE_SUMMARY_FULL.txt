1. HIGH-LEVEL DESCRIPTION
=========================

TS-Reasoner-v0 is a small, deterministic reasoning pipeline. It is designed to
show how a reasoning chain can be represented as a graph of constraint-bearing
steps, how local failures can be turned into inspectable tension issues, and
how repair suggestions can be attached to those failures.

The system is deliberately not a large language model. It does not download
weights, call an API, run a transformer, or claim general reasoning ability.
Instead, it uses hand-auditable standard-library heuristics over small
syllogistic templates. This makes the trace easy to inspect: every candidate
chain, step, claim, contradiction pair, tension score, and repair suggestion is
serializable to JSON.

The release is intended as a toy public artifact for showing the TS mapping:

  - TS-Core style substrate:
      Reasoning steps are graph nodes.
      Dependencies are support edges.
      Candidate chains are graph trajectories.

  - TensionLM style telemetry:
      Failures such as contradiction, unsupported conclusion, quantifier jump,
      missing premise, circular reasoning, and overconfidence are exposed as
      local tension rather than hidden inside a score.

  - Claim-Interaction Graph:
      Claims are extracted from steps with provenance back to the source step.
      Dependencies and contradiction pairs are exported.

  - Proof Ranker:
      Candidate chains are ranked by global tension and stability.
      The selected chain is the lowest-tension path under the toy scoring rules.


2. WHAT THIS RELEASE IS
=======================

This release is:

  - A standalone Python repository.
  - A deterministic toy reasoner.
  - A trace generator for small reasoning examples.
  - A public demonstration of constraint-graph reasoning telemetry.
  - A simple framework for showing how tension reduction can guide repair.
  - A release artifact with README, model card, examples, generated artifacts,

This release is not:

  - A trained model.
  - A TensionLM checkpoint.
  - A transformer wrapper.

  - A benchmark-grade reasoning evaluator.
  - A claim that regex heuristics solve natural-language reasoning.

4. CORE PACKAGE MODULES
=======================

4.1 ts_reasoner/types.py
------------------------

This module defines the shared dataclasses used throughout the system.

Main dataclasses:

  ReasoningStep
    Represents one node in a reasoning chain.
    Fields:
      step_id: stable identifier such as p1 or s1.
      text: human-readable step text.
      kind: premise, conclusion, or related step type.
      dependencies: list of support step IDs.
      confidence: toy confidence value.

  ReasoningChain
    Represents a candidate graph trajectory.
    Fields:
      chain_id: stable candidate identifier.
      question: input question.
      premises: normalized premise list.
      steps: ordered ReasoningStep objects.
      final_answer: final answer surfaced by the chain.
      generator: generator name.

  TensionIssue
    Represents a local failure or instability.
    Fields:
      issue_id: stable issue identifier.
      step_id: target step with tension.
      kind: issue type.
      severity: local numeric contribution.
      message: readable explanation.
      evidence: claim IDs or text fragments.
      suggestion: short repair direction.

  TensionScore
    Represents the local and global tension summary.
    Fields:
      chain_id: candidate chain ID.
      local_tension: mapping from step IDs to local scores.
      global_tension: aggregate instability score.
      issues: list of TensionIssue objects.
      stability: 1.0 minus global tension.

  Claim
    Represents an extracted claim with provenance.
    Fields:
      claim_id: source-step scoped claim identifier.
      text: original claim text.
      normalized: normalized claim string.
      source_step_id: step that produced the claim.
      subject: parsed subject.
      predicate: parsed predicate.
      quantifier: parsed quantifier if present.
      polarity: positive or negative.
      confidence: inherited from source step.
      dependencies: source step dependencies.

  CIGCheck
    Represents the Claim-Interaction Graph check output.
    Fields:
      chain_id: candidate chain ID.
      claims: extracted claims.
      dependencies: support edges by step ID.
      contradiction_pairs: conflicting claim ID pairs.
      unsupported_claim_ids: claims without sufficient premise support.
      circular_step_ids: self-supporting or circular steps.

  RepairSuggestion
    Represents an explicit repair proposal.
    Fields:
      repair_id: stable repair identifier.
      target_step_id: step to repair.
      issue_kind: issue being repaired.
      original_text: original step text.
      proposed_text: repaired or weakened text.
      rationale: why the repair reduces tension.
      expected_tension_delta: expected score movement.
      status: suggested by default.

  ReasonerOutput
    Represents the full output of one pipeline run.
    Fields:
      question, premises, candidates, selected_chain, tension_score, cig_check,
      repairs, final_answer, and trace.

Utility:

  to_jsonable()
    Recursively converts dataclasses into JSON-safe dictionaries and lists.

4.2 ts_reasoner/generator.py
----------------------------

This module creates deterministic candidate chains.

Important functions and classes:

  extract_relations(text)
    Uses a regular expression to extract simple relations of the form:
      "all A are B"
      "some A are B"
      "no A are B"

  infer_premises_from_question(question)
    Pulls simple if/then-style premises out of a question string. Example:
      Input:
        If all A are B and all B are C, are all A C?
      Inferred premises:
        all A are B
        all B are C

  infer_query_relation(question)
    Extracts the final asked relation from the question. It deliberately uses
    the final query-shaped match so the asked relation is not confused with an
    earlier premise.

  DeterministicHeuristicGenerator
    Generates small candidate sets. It normally emits:
      candidate_direct
      candidate_cautious
    For contradictory premise sets it can also emit:
      candidate_a_contradiction_aware

  TensionLMGenerator
    A future interface stub. It raises NotImplementedError because v0 does not
    include learned generation.

Candidate behavior:

  candidate_direct
    Tries to derive or assert the requested conclusion. It is useful because it
    can expose failures such as unsupported conclusions or quantifier jumps.

  candidate_cautious
    Gives a lower-tension answer when the universal claim is not forced. For a
    valid all/all transitive syllogism it still answers the derived universal.
    For invalid jumps or missing support it says the premises are insufficient.

  candidate_a_contradiction_aware
    Surfaces direct contradiction when a negative and positive relation collide.
    Its ID intentionally sorts before candidate_cautious when scores tie, so
    contradiction-aware output can be selected for contradiction tasks.


4.3 ts_reasoner/cig_checker.py
------------------------------

This module extracts a toy Claim-Interaction Graph.

Core behavior:

  - Extracts all simple relation claims from each step.
  - Extracts simple explicit negation claims like "A are not C".
  - Stores each claim with source-step provenance.
  - Stores dependency edges from step dependencies.
  - Detects contradiction pairs.
  - Detects unsupported conclusion claims.
  - Detects simple circular reasoning.

Contradiction detection:

  The checker treats a pair as contradictory when the subject and predicate
  match and one side is a negative universal or explicit negation while the
  other side is positive. Example:

    All A are C.
    No A are C.

Unsupported conclusion detection:

  A conclusion claim is treated as supported when:

    - the same claim appears in premise support, or
    - an all/all transitive bridge supports it.

  For example:

    All A are B.
    All B are C.
    Therefore all A are C.

  is treated as supported.

  But:

    Some A are B.
    All B are C.
    Therefore all A are C.

  is not treated as universal support.


4.4 ts_reasoner/ranker.py
-------------------------

This module scores local and global tension.

Main class:

  HeuristicTensionRanker

Issue kinds:

  contradiction
    A claim conflicts with another claim in the chain.

  unsupported_conclusion
    A conclusion claim is not supported by direct or transitive premise
    evidence.

  circular_reasoning
    A step depends on itself or uses language that assumes its own conclusion.

  quantifier_jump
    A conclusion upgrades "some" support into an "all" universal claim.

  missing_premise
    A conclusion is attempted with no explicit premises.

  overconfidence
    Surface language is more certain than support permits.

Score mechanics:

  - Each step begins with local tension 0.0.
  - Each issue adds severity to the target step.
  - Local tension is capped at 1.0.
  - Global tension is the sum of issue severities divided by the number of
    steps, capped at 1.0.
  - Stability is 1.0 minus global tension.

Severity values:

  contradiction: 0.55
  unsupported_conclusion: 0.35
  circular_reasoning: 0.45
  quantifier_jump: 0.50
  missing_premise: 0.40
  overconfidence: 0.20

Important caveat:

  These are transparent toy values, not calibrated probabilities.


4.5 ts_reasoner/repair.py
-------------------------

This module creates repair suggestions from tension issues.

Main class:

  TensionRepairer

Repair behavior:

  quantifier_jump
    Downgrades the first occurrence of "all" to "some" when possible, or
    otherwise states that the universal claim should be weakened.

  unsupported_conclusion
    Proposes an insufficiency answer unless a missing premise is supplied.

  contradiction
    Proposes surfacing the contradiction rather than forcing a conclusion.

  missing_premise
    Proposes requesting or adding premises.

  circular_reasoning
    Proposes replacing self-support with independent premise support.

  overconfidence
    Replaces some high-certainty words with weaker modal language.

The repairer does not secretly mutate the selected chain. It returns explicit
repair suggestions so the trace remains inspectable.

4.7 ts_reasoner/pipeline.py
---------------------------

This module ties the system together.

Main class:

  TSReasoner

Pipeline order:

  1. Normalize or infer premises.
  2. Generate candidate chains.
  3. Run CIG checks for each candidate.
  4. Score tension for each candidate.
  5. Select the chain with:
       lowest global tension,
       then highest stability,
       then lexical chain ID tie-break.
  6. Generate repair suggestions for the selected chain.
  7. Return ReasonerOutput.

Convenience function:

  run_reasoner(question, premises=None)

Trace object includes:

  pipeline:
    "TS-Reasoner-v0"

  generator:
    "DeterministicHeuristicGenerator"

  selection:
    selected chain ID and selection criterion.

  candidate_scores:
    chain ID, global tension, stability, and issue kinds for each candidate.

  graph_view:
    selected graph nodes, dependency edges, claim count, and contradiction count.

13. DESIGN DECISIONS
====================

13.1 Standard library only
--------------------------

The release avoids external dependencies so it can run in a clean Python
environment. JSON writing, dataclasses, argparse, regex parsing, and unit tests
all use the standard library.


13.2 Deterministic heuristics
-----------------------------

The generator and ranker are deterministic. This makes generated traces stable
and makes tests reliable.


13.3 Candidate plurality
------------------------

The pipeline generates more than one chain so the ranker has something to
compare. This matters for TS framing because reasoning is treated as a graph
trajectory selection problem, not a single hidden answer.


13.4 Cautious chain
-------------------

The cautious chain is allowed to say that premises are insufficient. This keeps
invalid universal claims from being selected when the direct chain has high
tension.


13.5 Contradiction-aware chain
------------------------------

When contradictory premises are present, the generator can produce a candidate
that surfaces graph instability. Its ID sorts before candidate_cautious in a
tie so contradiction tasks can select contradiction-aware output when tension
scores are tied or close.


13.6 Explicit repair suggestions
--------------------------------

Repairs are not hidden rewrites. The repairer returns structured
RepairSuggestion objects with target step, issue kind, original text, proposed
text, rationale, and expected tension delta.


13.7 Toy metrics only
---------------------

The demo summary is framed as toy-scope receipts, not benchmark performance.
This preserves the distinction between observed local behavior and broader
research claims.


14. CURRENT BEHAVIOR BY FAILURE TYPE
====================================

14.1 Quantifier jump
--------------------

Pattern:

  Some A are B.
  All B are C.
  Therefore all A are C.

Detected as:

  quantifier_jump
  unsupported_conclusion

Repair style:

  Downgrade the universal claim or answer that the universal does not follow.


14.2 Direct contradiction
-------------------------

Pattern:

  All A are C.
  No A are C.

Detected as:

  contradiction

Repair style:

  Surface contradiction and avoid forced stable conclusion.


14.3 Missing premise
--------------------

Pattern:

  Are all A C?

Detected as:

  missing_premise

Repair style:

  Request premises or return insufficiency.


14.4 Unsupported conclusion
---------------------------

Pattern:

  Conclusion relation has no direct premise support and no recognized all/all
  transitive bridge.

Detected as:

  unsupported_conclusion

Repair style:

  Add missing premise or weaken the answer to insufficiency.


14.5 Circular reasoning
-----------------------

Pattern:

  A step depends on itself, or uses simple circular language such as "because
  the conclusion" or "assume the conclusion".

Detected as:

  circular_reasoning

Repair style:

  Replace circular support with independent premise support.


14.6 Overconfidence
-------------------

Pattern:

  Surface text uses certainty markers such as "definitely", "certainly",
  "always", "obviously", or "must be true" while support is weak.

Detected as:

  overconfidence

Repair style:

  Use calibrated language tied to available support.


15. LIMITATIONS
===============

The most important limitations are:

  - Regex extraction is narrow.
  - Only simple quantifiers are handled: all, some, no.
  - Relation parsing expects short "X are Y" style claims.
  - The system is not a full natural-language parser.
  - The system is not a formal proof engine.
  - It cannot handle complex nested conditionals.
  - It cannot handle most real-world ambiguity.
  - It does not learn.
  - It does not call TensionLM.
  - It does not call a trained proof ranker.
  - Numeric tension values are heuristic and not calibrated.
  - The demo set has only five toy tasks.
  - The eval summary is a release receipt, not a benchmark.

18. FINAL STATE
===============

Final state summary:

  TS-Reasoner-v0 was implemented as a compact, inspectable, standard-library
  Python system. The core substrate is a set of candidate reasoning chains with
  graph-like step dependencies. The CIG checker extracts claims and support
  edges. The ranker turns local graph failures into tension. The repairer turns
  high-tension issues into explicit proposed relaxations. The pipeline selects
  the lowest-tension candidate and exports a full trace.

  Verification passed with unittest. The demo artifacts were regenerated from
  the actual code. The release explicitly separates observed toy behavior from
  broader TS/TensionLM research hypotheses.
